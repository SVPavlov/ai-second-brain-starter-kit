# Wrap up session  ·  "Wrap up my session"

The end-of-session handoff. Five minutes of capture now saves the "waar was ik ook alweer?" half hour tomorrow.

## Steps

1. **Harvest from the session.** From the conversation and the files touched, extract: what got DONE, what is NEXT (concrete steps, not themes), what is BLOCKED and on whom, and any decision that was made along the way. Re-read the touched files rather than trusting memory; be specific, one line per item.
2. **Decisions get their own record.** If the session contains a real decision (chosen between options, hard to reverse), suggest running record-decision for it; do not bury it in the log.
3. **Write the handoff where the routine will find it:**
   - Open actions → `03-tasks.md`, each with an owner and a date (skip items already on the list).
   - "Next up" and blockers → the relevant section of `02-command-center.md`, so tomorrow's daily brief opens with them.
   - Client-related next steps → also the client's `_index.md` hub, so the next client-context briefing knows.
   - One summary line → `04-log.md` (date, wrap-up, what was done, files touched).
4. **Unresolved questions** that only the owner can answer go to `05-synthesis.md` (asks, never guesses), not into the handoff as fake certainty.
5. **Confirm** with a short readback: done / next / blocked, and where each landed.

## Guardrails

- The handoff records what happened; it does not invent progress. If something failed or stayed unfinished, write that, plainly.
- No duplicates: check `03-tasks.md` and the command center before adding.
- Keep it scannable. If the wrap-up needs more than ~15 lines total, it is summarizing too little or the session was several sessions in a trench coat; say so.
